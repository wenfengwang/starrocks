# 为 StarRocks 贡献力量

我们热烈欢迎每个人为 StarRocks 做出贡献。为 StarRocks 做出贡献不仅限于编写代码。以下，我们列出了为我们的社区做出贡献的不同途径。

|报告 Bug|您可以[提交 issue](https://github.com/StarRocks/starrocks/issues/new/choose) 来报告 StarRocks 的 Bug。您也可以点击您正在阅读的文档站点页面右上角的 `Feedback` 来报告 Bug。|
|---|---|
|**贡献代码**|您可以通过修复 Bug 或实现新功能来贡献您的代码。|
|**贡献测试用例**|您可以贡献您的测试用例。|
|**帮助审查代码**|如果您是 StarRocks 的活跃贡献者或 Committer，您可以帮助我们审查 Pull Requests (PRs)。|
|**贡献文档**|StarRocks 社区维护着大量中英文文档。您可以通过修复文档 Bug 或提出新的内容来贡献文档更改。|
|**帮助 StarRocks 用户**|您可以帮助社区中遇到困难的新用户。|
|**传播 StarRocks 的信息**|您可以撰写文章或发表演讲来帮助向全世界传播我们的技术。|

> **注意**
> 要贡献文档，请记得使用 `git commit -s` 来**签署**您的提交。否则，开发者证书原产地 (DCO) 检查将失败，PR 可能会被阻止。此外，请在您的 PR 标题前加上 `[Doc]` 前缀，并勾选 **Doc** 复选框。

## 社区角色

![社区角色](../assets/contri-1.png)

我们鼓励每个人参与 StarRocks 项目。任何人都可以通过参与讨论新功能、项目路线图、架构，甚至报告您所面临的问题来产生影响。

以下列出的角色是参与社区的几种可能方式，并定义了每个角色所期望的内容。

## 用户组

### 参与者

参与者积极参与社区并致力于让 StarRocks 更好地服务于每个人。

作为参与者，您可以在 GitHub 上提交问题报告、贡献测试用例、翻译/修改文档、在论坛或社区中帮助回答用户问题、参与活动、分享您的 StarRocks 使用经验、在 GitHub 上给 StarRocks 点赞等等。

期望和责任：

- 遵循社区的 [行为守则](https://github.com/StarRocks/starrocks/blob/main/CODE_OF_CONDUCT.md)。

- 参与讨论和社区活动。

- 向社区提供反馈，让其他人知道哪些是有效/无效的。

- 提出改进建议。

### 明星贡献者

StarRocks 社区明星贡献者是一群热情的社区布道者和先锋，他们精通 StarRocks 技术。他们乐于在社区中分享最新的发展和产品，他们的热情推动了社区的进步和发展。

#### 好处

产品：

- 提前接触新产品/功能。

- 与项目核心团队参与独家会议。

社区：

- 在网站上被列为 StarRocks 明星贡献者。

- 独家奖品和徽章。

活动：

- 免费参加 StarRocks 的活动。

- 报销参加活动的差旅和住宿费用。

- VIP 座位和 VIP 活动。

#### 要求

1. 至少发表过 3 篇与 StarRocks 相关的原创且高质量的技术内容（包括文章、教程、视频等）。
2. 至少作为演讲者参加过 2 次线上/线下技术分享会。
3. 在 Slack 和其他渠道上帮助回答用户问题。

#### 其他要求

1. StarRocks 明星贡献者必须遵守 StarRocks 社区 [行为守则](https://github.com/StarRocks/starrocks/blob/main/CODE_OF_CONDUCT.md)。
2. StarRocks 明星贡献者的资格有效期为一年，每年必须根据过去一年的贡献重新申请。

## 开发者组

### 贡献者

每个做出贡献的人都可以成为 StarRocks 贡献者。当新贡献者需要帮助时，成员将提供指导和支持。

#### 如何成为贡献者？

- 在 StarRocks 的任何公共仓库中有 1 个合并的 PR。

作为贡献者，我们期望您：

- 积极参与 StarRocks 项目的开发。

- 参与社区活动（聚会、黑客马拉松等）。

- 学习并帮助他人学习 StarRocks 相关技术。

#### 特权

- 被列为 StarRocks 贡献者。

- 获得 StarRocks 贡献者电子证书。

### 积极贡献者

积极贡献者是那些为 StarRocks 做出杰出贡献并持续承诺的贡献者。他们通过贡献代码、改进文档和帮助他人积极参与社区。

#### 如何成为积极贡献者？

- 有 5 个合并的 PR 或修复了重大 Bug。

- 参与超过 5 次代码审查。

- 积极参与社区活动，如线上/线下聚会和社区讨论。

#### 责任和特权

- 参与社区会议和讨论。

- 指导和支持新贡献者。

- 被列为 StarRocks 积极贡献者。

- 获得 StarRocks 积极贡献者电子证书。

### Committer

Committer 是从积极贡献者中晋升而来。

他们有权限将 PR 合并到主分支，并负责 StarRocks 的规划和维护。他们也是在社区中分享知识的积极成员。

#### 如何成为 Committer？

- 深入理解 StarRocks 的原则和未来计划。

- 能够及时处理项目中出现的各种问题。

- 领导重大开发，编写和修订相关文档。

- 获得至少两个 PMC 的提名并通过投票。

#### 责任和特权

- 指导和支持社区中的其他成员。

- 确保子项目持续健康发展。

- 获得对 StarRocks 仓库的写入权限（具体待定）。

- 被列为 StarRocks Committer。

- 获得 StarRocks Committer 电子证书。

### PMC

PMC 是从 Committer 中晋升而来。他们有权限将 PR 合并到主分支，并负责 StarRocks 的规划和维护。他们也是在社区中分享知识的积极成员。

#### 如何成为 PMC？

- 深入理解 StarRocks 的原则和清晰了解 StarRocks 的未来规划。

- 能够及时处理项目问题。

- 领导项目开发和迭代，引导项目的整体方向。

- 获得至少两个 PMC 的提名并通过投票。

#### 责任和特权

- 指导和支持社区中的其他成员。

- 确保项目持续健康，如代码质量和测试覆盖率。

- 制定并批准技术设计决策。

- 定义里程碑和发布版本。

- 投票并提拔新的 Committer 和 PMC。

- 被列为 StarRocks PMC。

- 获得 StarRocks PMC 电子证书。

## 贡献流程

如果您不知道如何开始，这是我们建议的贡献流程。此流程旨在帮助您减少学习曲线并更高效地合并您的 Pull Requests！

1. 签署 [贡献者许可协议 (CLA)](https://cla-assistant.io/StarRocks/starrocks)。

2. 通过创建一个 GitHub [issue](https://github.com/StarRocks/starrocks/issues)，或在 [Slack 频道](https://join.slack.com/t/starrocks/shared_invite/zt-1d0cs3cs2-7eL5Q0cGoBofhyIOx7pLfQ) 上提问来开始讨论（除非更改微不足道）。

   在动手编码之前，您应该在 issue 正文中评论，并通知维护者将您希望解决的 issue 分配给您。建议在 issue 正文中分享您解决问题的计划。

   - 这一步骤可以帮助您找到可能的合作者和审阅者。
   - 这个更改是否会与正在进行的其他更改发生冲突？如果是，请与其他人合作以尽量减少影响。
   - 这个更改是否重大？如果是，请与其他人合作将更改分解为更小的步骤。

3. 实施更改。

-    如果更改较大，请将其分解为更小的 PR。

-    必要时包括测试和文档。

4. 创建一个 GitHub [Pull Request](https://github.com/StarRocks/starrocks/pulls)：

   在 StarRocks 社区中，我们在贡献代码时遵循 fork-and-merge GitHub 工作流程。

   - 在您的 GitHub 账户中 fork StarRocks。
   - 克隆这个 fork 的仓库到您的计算机。
   - 基于您期望贡献的分支检出一个新分支。
   - 将您的代码更改提交到新分支。
   - 将带有代码更改的分支推送到 GitHub。
   - 创建一个 PR 来提交您的代码更改。建议在一个 PR 中提交一个 commit。您可以在提交 PR 时参照 [PR 模板](https://github.com/StarRocks/starrocks/blob/main/.github/PULL_REQUEST_TEMPLATE.md)。
   - 确保 Pull Request 通过 CI 中的测试。

5. 至少需要 2 名审阅者进行审阅：

-    对于 StarRocks 项目，我们要求至少有 2 个来自审阅者（Committer）的 LGTM 才能合并 Pull Request。

-    这通常会在几天内完成，但如果更改重大、复杂，或关键审阅者不在，可能需要更长时间。（请随时在 Pull Request 中提醒审阅者）。

6. 维护者在最终更改被接受后合并 Pull Request。
