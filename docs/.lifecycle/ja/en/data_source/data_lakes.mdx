# データレイクハウス

<DataLakeIntro />

## 主要なコンセプト

- オープンデータフォーマット: JSON、Parquet、Avro などの様々なデータタイプをサポートし、構造化データと非構造化データの保存および処理を容易にします。
- メタデータ管理: 共有メタデータレイヤーを実装し、Iceberg テーブルフォーマットなどのフォーマットを使用してデータを効率的に整理し管理します。
- 多様なクエリエンジン: Presto や Spark の強化バージョンなど、様々な分析や AI ユースケースに対応するための複数のエンジンを組み込んでいます。
- ガバナンスとセキュリティ: データセキュリティ、プライバシー、コンプライアンスのための堅牢な組み込みメカニズムを特徴とし、データの整合性と信頼性を保証します。

## データレイクハウスアーキテクチャのメリット

- 柔軟性とスケーラビリティ: 多様なデータタイプをシームレスに管理し、組織のニーズに応じてスケールアップできます。
- コスト効率: 伝統的な方法に比べて、データ保存と処理のための経済的な代替案を提供します。
- 強化されたデータガバナンス: データのコントロール、管理、整合性を改善し、信頼性の高い安全なデータハンドリングを実現します。
- AI と分析のための準備: 機械学習や AI 駆動のデータ処理を含む複雑な分析タスクに適しています。

## StarRocks のアプローチ

考慮すべき主要な要素は以下の通りです:

- カタログサービスやメタデータサービスとの統合を標準化すること
- コンピュートノードの伸縮性
- 柔軟なキャッシュメカニズム

---

## カタログ

StarRocks には内部カタログと外部カタログの2種類があります。内部カタログには StarRocks データベース内に保存されているデータのメタデータが含まれます。外部カタログは Hive、Iceberg、Delta Lake、Hudi が管理するデータなど、外部に保存されたデータを操作するために使用されます。その他の外部システムについては、ページ下部の「詳細情報」セクションにリンクがあります。

## コンピュートノード (CN) のスケーリング

ストレージとコンピュートの分離により、スケーリングの複雑さが減少します。StarRocks のコンピュートノードはローカルキャッシュのみを保持しているため、負荷に応じてノードを追加または削除することができます。

## データキャッシュ

コンピュートノード上のキャッシュはオプションです。コンピュートノードが急速に変化する負荷パターンに基づいてすぐに起動および停止する場合や、クエリが最新のデータにのみ頻繁に行われる場合は、データをキャッシュすることは意味がありません。

## 実践を通じて学ぶ

Docker ベースのレイクハウスを試してみてください。この[データレイクハウスチュートリアル](../quick_start/iceberg.md)では、Iceberg と MinIO を使用してデータレイクを構築し、StarRocks コンテナをクエリエンジンとして接続します。提供されたデータセットをロードするか、または自分のデータセットを追加してください。
