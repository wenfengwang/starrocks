## 開始する前に

### ソースデータを準備する

StarRocksに読み込むソースデータがS3バケットに適切に格納されていることを確認してください。また、データとデータベースの場所を考慮することも重要です。なぜなら、バケットとStarRocksクラスターが同じリージョンにある場合、データ転送コストが大幅に削減されるからです。

このトピックでは、S3バケット内のサンプルデータセット`s3://starrocks-datasets/user_behavior_ten_million_rows.parquet`を提供しています。このオブジェクトはAWSの認証ユーザーならば読み取りが可能ですので、適切な資格情報を使用してそのデータセットにアクセスできます。

### 権限を確認する

<InsertPrivNote />

### 接続詳細を収集する

このトピックの例では、IAMユーザーベースの認証が使用されています。AWS S3からデータを読み込むためのアクセス権を持っていることを確認するために、[IAMユーザーベースの認証の準備](../integrations/authenticate_to_aws_resources.md)をお読みになり、適切に構成された[IAMポリシー](../reference/aws_iam_policies.md)を持つIAMユーザーを作成する手順に従ってください。

要するに、IAMユーザーベースの認証を実施する場合は、以下のAWSリソースに関する情報を収集する必要があります：

- データを格納しているS3バケット。
- バケット内の特定のオブジェクトにアクセスする場合は、S3オブジェクトキー（オブジェクト名）。S3オブジェクトがサブフォルダに格納されている場合は、オブジェクトキーにプレフィックスを含めることができます。
- S3バケットが属するAWSリージョン。
- アクセス認証として使用するアクセスキーとシークレットキー。

利用可能なすべての認証方法に関する情報については、[AWSリソースへの認証](../integrations/authenticate_to_aws_resources.md)をご参照ください。

## INSERT+FILES()を使用する

このメソッドはv3.1から利用可能であり、現在はParquetおよびORCファイルフォーマットのみをサポートしています。

### INSERT+FILES()の利点

[`FILES()`](../sql-reference/sql-functions/table-functions/files.md)は、指定したパス関連のプロパティに基づいてクラウドストレージに保存されているファイルを読み取り、ファイル内のデータのテーブルスキーマを推測し、そのデータをデータ行として返します。

`FILES()`を使用すると、次のことができます：

- [SELECT](../sql-reference/sql-statements/data-manipulation/SELECT.md)を使用してS3からデータを直接クエリします。
- [CREATE TABLE AS SELECT](../sql-reference/sql-statements/data-definition/CREATE_TABLE_AS_SELECT.md)（CTAS）を使用して新しいテーブルを作成し、データをロードします。
- [INSERT](../sql-reference/sql-statements/data-manipulation/INSERT.md)を使用して既存のテーブルにデータをロードします。

### 典型的な例

#### SELECTを使用してS3から直接クエリする

SELECT+`FILES()`を使用してS3から直接クエリを実行すると、テーブルを作成する前にデータセットの内容を確認できます。例：

- データを保存せずにデータセットのプレビューを表示する。
- 最小値と最大値をクエリし、使用するデータ型を決定する。
- `NULL`値を確認する。

次の例では、サンプルデータセット`s3://starrocks-datasets/user_behavior_ten_million_rows.parquet`をクエリしています：

```SQL
SELECT * FROM FILES
(
    "path" = "s3://starrocks-datasets/user_behavior_ten_million_rows.parquet",
    "format" = "parquet",
    "aws.s3.region" = "us-east-1",
    "aws.s3.access_key" = "AAAAAAAAAAAAAAAAAAAA",
    "aws.s3.secret_key" = "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB"
)
LIMIT 3;
```

> **注意**
>
> 上記のコマンドで`AAA`と`BBB`を自身の資格情報に置き換えてください。有効な`aws.s3.access_key`と`aws.s3.secret_key`ならば、どんなAWSの認証ユーザーも使用できます。

システムは次のクエリ結果を返します：

```Plaintext
+--------+---------+------------+--------------+---------------------+
| UserID | ItemID  | CategoryID | BehaviorType | Timestamp           |
+--------+---------+------------+--------------+---------------------+
| 543711 |  829192 |    2355072 | pv           | 2017-11-27 08:22:37 |
| 543711 | 2056618 |    3645362 | pv           | 2017-11-27 10:16:46 |
| 543711 | 1165492 |    3645362 | pv           | 2017-11-27 10:17:00 |
+--------+---------+------------+--------------+---------------------+
```

> **注意**
>
> 上記の結果で返される列名はParquetファイルによって提供されていることに留意してください。

#### CTASを使用してテーブルを作成およびロードする

これは前の例の続きです。前のクエリをCREATE TABLE AS SELECT (CTAS)でラップして、スキーマ推測を使用してテーブルを自動的に作成し、データをロードします。`FILES()`テーブル関数をParquetファイルで使用する場合、テーブル作成のために列名とタイプが必要ではありません。

> **注意**
>
> スキーマ推測を使用する場合、CREATE TABLEの構文はレプリカ数を設定することを許可していませんので、テーブルを作成する前にレプリカ数を設定してください。以下は、シングルレプリカのシステムに対する例です：
>
> ```SQL
> ADMIN SET FRONTEND CONFIG ('default_replication_num' = "1");
> ```

データベースを作成し、そのデータベースを使用します：

```SQL
CREATE DATABASE IF NOT EXISTS mydatabase;
USE mydatabase;
```

CTASを使用して新しいテーブルを作成し、サンプルデータセット`s3://starrocks-datasets/user_behavior_ten_million_rows.parquet`のデータをそのテーブルにロードします：

```SQL
CREATE TABLE user_behavior_inferred AS
SELECT * FROM FILES
(
    "path" = "s3://starrocks-datasets/user_behavior_ten_million_rows.parquet",
    "format" = "parquet",
    "aws.s3.region" = "us-east-1",
    "aws.s3.access_key" = "AAAAAAAAAAAAAAAAAAAA",
    "aws.s3.secret_key" = "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB"
);
```

> **注意**
>
> 上記のコマンドで`AAA`と`BBB`を自身の資格情報に置き換えてください。有効な`aws.s3.access_key`と`aws.s3.secret_key`ならば、どんなAWSの認証ユーザーも使用できます。

テーブルを作成した後、[DESCRIBE](../sql-reference/sql-statements/Utility/DESCRIBE.md)を使用してそのスキーマを表示できます：

```SQL
DESCRIBE user_behavior_inferred;
```

システムは次のクエリ結果を返します：

```Plaintext
+--------------+------------------+------+-------+---------+-------+
| Field        | Type             | Null | Key   | Default | Extra |
+--------------+------------------+------+-------+---------+-------+
| UserID       | bigint           | YES  | true  | NULL    |       |
| ItemID       | bigint           | YES  | true  | NULL    |       |
| CategoryID   | bigint           | YES  | true  | NULL    |       |
| BehaviorType | varchar(1048576) | YES  | false | NULL    |       |
| Timestamp    | varchar(1048576) | YES  | false | NULL    |       |
+--------------+------------------+------+-------+---------+-------+
```

推測されたスキーマを手で作成したスキーマと比較してください：

- データ型
- NULL可能
- キーフィールド

宛に、宛先テーブルのスキーマをよりよく制御し、クエリパフォーマンスを向上させるために、本番環境ではテーブルスキーマを手で指定することを推奨します。

データが正常にロードされたことを確認するために、テーブルをクエリします。例：

```SQL
SELECT * from user_behavior_inferred LIMIT 3;
```

以下のクエリ結果が返され、データが正常にロードされたことが示されます：

```Plaintext
+--------+--------+------------+--------------+---------------------+
| UserID | ItemID | CategoryID | BehaviorType | Timestamp           |
+--------+--------+------------+--------------+---------------------+
|     58 | 158350 |    2355072 | pv           | 2017-11-27 13:06:51 |
|     58 | 158590 |    3194735 | pv           | 2017-11-27 02:21:04 |
|     58 | 215073 |    3002561 | pv           | 2017-11-30 10:55:42 |
+--------+--------+------------+--------------+---------------------+
```

#### INSERTを使用して既存のテーブルにロードする

挿入するテーブルをカスタマイズしたい場合、たとえば以下のようなことが挙げられます：

- カラムのデータ型、NULL設定、またはデフォルト値
- キータイプとカラム
- データパーティショニングとバケット分割

> **注意**
>
> 以下はステップバイステップの手順になります:  
> 1. データがどのように使用され、列の内容を知ることが、最も効率的なテーブル構造を作成するために必要となります。このトピックでは、テーブル設計には触れません。テーブル設計については、[テーブルタイプ](../table_design/StarRocks_table_design.md)を参照してください。

> この例では、クエリされるテーブルとParquetファイルのデータを知識に基づいてテーブルを作成しています。Parquetファイルのデータ知識は、S3でファイルを直接クエリすることで取得できます。

> - S3のデータセットのクエリにより、`Timestamp`列が`datetime`データ型に一致するデータを含んでいることが示されたため、列のタイプは次のDDLで指定されています。
> - S3のデータをクエリすることで、データセットに`NULL`値がないことがわかるため、DDLではどの列もnullableとして設定されていません。
> - 期待されるクエリタイプの知識に基づいて、ソートキーとバケット化された列は列`UserID`に設定されています。このデータのユースケースは異なる場合がありますので、ソートキーとして`UserID`のほかに`ItemID`を使用することもあります。

> データベースを作成し、それに切り替えます。

```SQL
CREATE DATABASE IF NOT EXISTS mydatabase;
USE mydatabase;
```

> 手動でテーブルを作成します（AWS S3から読み込むParquetファイルと同じスキーマを持つことをお勧めします）：

```SQL
CREATE TABLE user_behavior_declared
(
    UserID int(11),
    ItemID int(11),
    CategoryID int(11),
    BehaviorType varchar(65533),
    Timestamp datetime
)
ENGINE = OLAP 
DUPLICATE KEY(UserID)
DISTRIBUTED BY HASH(UserID)
PROPERTIES
(
    "replication_num" = "1"
);
```

> テーブルを作成した後、INSERT INTO SELECT FROM FILES()でそのテーブルにデータをロードできます：

```SQL
INSERT INTO user_behavior_declared
SELECT * FROM FILES
(
    "path" = "s3://starrocks-datasets/user_behavior_ten_million_rows.parquet",
    "format" = "parquet",
    "aws.s3.region" = "us-east-1",
    "aws.s3.access_key" = "AAAAAAAAAAAAAAAAAAAA",
    "aws.s3.secret_key" = "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB"
);
```

> **注意**
>
> 上記のコマンドの`AAA`と`BBB`をご自身の資格情報に置き換えてください。任意の有効な`aws.s3.access_key`と`aws.s3.secret_key`を使用できますが、そのオブジェクトはAWS認証ユーザーによって読み取られます。

> ロードが完了したら、そのデータが正常にロードされていることを確認するためにテーブルをクエリできます。例：

```SQL
SELECT * from user_behavior_declared LIMIT 3;
```

> 次のクエリの結果が返され、データが正常にロードされていることが示されます：

## ブローカーロードを使用

非同期のブローカーロードプロセスは、S3への接続、データの取得、およびStarRocksへのデータの保存を処理します。

このメソッドはParquet、ORC、およびCSVファイルフォーマットをサポートしています。

### ブローカーロードの利点

- ブローカーロードは、ロード中の[データ変換](../loading/Etl_in_loading.md)および[UPSERTおよびDELETE操作などのデータ変更](../loading/Load_to_Primary_Key_tables.md)をサポートしています。
- ブローカーロードはバックグラウンドで実行され、ジョブが続行するためにクライアントが接続を維持する必要はありません。
- ブローカーロードはデフォルトタイムアウトが4時間に及ぶ長時間実行ジョブに適しています。
- ParquetおよびORCファイルフォーマットに加えて、ブローカーロードはCSVファイルをサポートしています。

### データフロー

![ブローカーロードのワークフロー](../assets/broker_load_how-to-work_en.png)

1. ユーザーがロードジョブを作成します。
2. フロントエンド（FE）がクエリプランを作成し、プランをバックエンドノード（BE）に配布します。
3. BEがソースからデータを取得し、データをStarRocksにロードします。

### 典型的な例

データセット `s3://starrocks-datasets/user_behavior_ten_million_rows.parquet`から`user_behavior`テーブルにデータをロードするブローカーロードジョブを開始し、データのロードの進行状況および成功を検証します。

#### データベースとテーブルの作成

データベースを作成し、それに切り替えます：

```SQL
CREATE DATABASE IF NOT EXISTS mydatabase;
USE mydatabase;
```

手動でテーブルを作成します（AWS S3から読み込むParquetファイルと同じスキーマを持つことをお勧めします）：

```SQL
CREATE TABLE user_behavior
(
    UserID int(11),
    ItemID int(11),
    CategoryID int(11),
    BehaviorType varchar(65533),
    Timestamp datetime
)
ENGINE = OLAP 
DUPLICATE KEY(UserID)
DISTRIBUTED BY HASH(UserID)
PROPERTIES
(
    "replication_num" = "1"
);
```

#### ブローカーロードを開始

次のコマンドを実行して、サンプルデータセット`s3://starrocks-datasets/user_behavior_ten_million_rows.parquet`から`user_behavior`テーブルにデータをロードするブローカーロードジョブを開始します：

```SQL
LOAD LABEL user_behavior
(
    DATA INFILE("s3://starrocks-datasets/user_behavior_ten_million_rows.parquet")
    INTO TABLE user_behavior
    FORMAT AS "parquet"
 )
 WITH BROKER
 (
    "aws.s3.enable_ssl" = "true",
    "aws.s3.use_instance_profile" = "false",
    "aws.s3.region" = "us-east-1",
    "aws.s3.access_key" = "AAAAAAAAAAAAAAAAAAAA",
    "aws.s3.secret_key" = "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB"
 )
PROPERTIES
(
    "timeout" = "72000"
);
```

> **注意**
>
> 上記のコマンドの`AAA`と`BBB`をご自身の資格情報に置き換えてください。任意の有効な`aws.s3.access_key`と`aws.s3.secret_key`を使用できますが、そのオブジェクトはAWS認証ユーザーによって読み取られます。

このジョブには以下の4つの主要セクションがあります：
- `LABEL`：ロードジョブの状態をクエリする際に使用される文字列。
- `LOAD` 宣言：ソースURI、ソースデータフォーマット、およびディスティネーションテーブル名。
- `BROKER`：ソースの接続詳細。
- `PROPERTIES`：ロードジョブに適用するタイムアウト値およびその他のプロパティ。

詳細な構文とパラメータの説明については、[BROKER LOAD](../sql-reference/sql-statements/data-manipulation/BROKER_LOAD.md)を参照してください。

## パイプの使用

v3.2から、StarRocksはParquetおよびORCファイルフォーマットのみをサポートするパイプロードメソッドを提供します。

### パイプの利点

パイプは連続データロードと大規模データロードに適しています：

- **マイクロバッチでの大規模データロードは、データエラーに起因するリトライコストを削減します。**

  パイプのおかげで、StarRocksは合計データ量が多い大量のデータファイルを効率的にロードできます。パイプは自動的にファイルを分割し、ロードジョブを小さな連続的なタスクに分割します。このアプローチにより、1つのファイルのエラーがジョブ全体に影響を与えないようになります。各ファイルのロードステータスはパイプによって記録され、エラーを含むファイルを簡単に特定および修正できます。データエラーによるリトライの必要性を最小限に抑えることで、コストを削減します。

- **連続データロードは人手を削減します。**

  パイプは、特定の場所に新しいまたは更新されたデータファイルを書き込み、それらのファイルから新しいデータをStarRocksに連続してロードします。`"AUTO_INGEST" = "TRUE"`が指定されたパイプジョブを作成した後、指定されたパスに保存されているファイルの変更を常に監視し、データファイルから新しいまたは更新されたデータを自動的にStarRocksテーブルにロードします。

さらに、パイプは重複データロードを防ぐためにファイルの一意性を確認します。ロードプロセス中、パイプはファイル名とダイジェストに基づいて各データファイルの一意性をチェックします。特定のファイル名とダイジェストを持つファイルがすでにパイプジョブによって処理されている場合、パイプジョブは同じファイル名とダイジェストを持つすべての後続ファイルをスキップします。AWS S3などのオブジェクトストレージでは、ファイルダイジェストとして`ETag`が使用されます。

各データファイルのロードステータスは`information_schema.pipe_files`ビューに記録され、そのビューに関連付けられたパイプジョブが削除された後、そのジョブでロードされたファイルに関するレコードも削除されます。

### パイプとINSERT+FILES()の違い

パイプジョブは各データファイルのサイズと行数に基づいて1つ以上のトランザクションに分割されます。ユーザーはロードプロセス中に中間結果をクエリできます。これに対して、INSERT+`FILES()`ジョブは単一のトランザクションとして処理され、ユーザーはロードプロセス中にデータを表示できません。

### ファイルのロード順序

各パイプジョブについて、StarRocksはファイルキューを維持し、マイクロバッチとしてデータファイルを取得およびロードします。パイプはデータファイルがアップロードされた順序と同じ順序でロードされることを保証しません。したがって、新しいデータが古いデータよりも先にロードされることがあります。

### 典型的な例

#### データベースとテーブルを作成

データベースを作成し、それに切り替えます：

```SQL
CREATE DATABASE IF NOT EXISTS mydatabase;
USE mydatabase;
```

手動でテーブルを作成します（Parquetファイルのスキーマと同じスキーマを持つことをお勧めします）：

```SQL
CREATE TABLE user_behavior_replica
(
    UserID int(11),
    ItemID int(11),
    CategoryID int(11),
    BehaviorType varchar(65533),
    Timestamp datetime
)
ENGINE = OLAP 
DUPLICATE KEY(UserID)
DISTRIBUTED BY HASH(UserID)
PROPERTIES
(
    "replication_num" = "1"
);
```

#### パイプジョブを開始

次のコマンドを実行して、サンプルデータセット `s3://starrocks-datasets/user_behavior_ten_million_rows.parquet` から `user_behavior_replica` テーブルにデータをロードするパイプジョブを開始します：

```SQL
CREATE PIPE user_behavior_replica
PROPERTIES
(
    "AUTO_INGEST" = "TRUE"
)
AS
INSERT INTO user_behavior_replica
SELECT * FROM FILES
(
    "path" = "s3://starrocks-datasets/user_behavior_ten_million_rows.parquet",
    "format" = "parquet",
    "aws.s3.region" = "us-east-1",
    "aws.s3.access_key" = "AAAAAAAAAAAAAAAAAAAA",
    "aws.s3.secret_key" = "BBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBBB"
); 
```

> **注記**
>
> 上記のコマンドで `AAA` と `BBB` の資格情報を置き換えてください。オブジェクトは、認証された AWS ユーザーであれば誰でも読み取り可能なので、有効な `aws.s3.access_key` と `aws.s3.secret_key` を使用できます。

このジョブには4つの主要なセクションがあります。

- `pipe_name`: パイプの名前。パイプの名前は、それが属するデータベース内で一意である必要があります。
- `INSERT_SQL`: 指定したソースデータファイルから宛先テーブルにデータをロードするために使用される INSERT INTO SELECT FROM FILES ステートメント。
- `PROPERTIES`: パイプを実行する方法を指定するオプションのパラメータのセット。これには `AUTO_INGEST`、`POLL_INTERVAL`、`BATCH_SIZE`、`BATCH_FILES` などが含まれます。これらのプロパティは、`"key" = "value"` の形式で指定します。

詳細な構文とパラメータの説明については、[CREATE PIPE](../sql-reference/sql-statements/data-manipulation/CREATE_PIPE.md) を参照してください。

#### ロードの進行状況を確認する

- [SHOW PIPES](../sql-reference/sql-statements/data-manipulation/SHOW_PIPES.md) を使用して Pipe ジョブの進行状況をクエリします。

  ```SQL
  SHOW PIPES;
  ```

  複数のロードジョブを送信した場合は、ジョブに関連する `NAME` でフィルタリングできます。例:

  ```SQL

     LAST_ERROR: NULL

  ```

  ```SQL
  SELECT * FROM information_schema.pipes;
  ```

  複数のロードジョブを送信した場合は、ジョブに関連する `PIPE_NAME` でフィルタリングできます。例:


#### Pipe ジョブの管理

作成したパイプを変更、中断、再開、削除したり、特定のデータファイルのロードを再試行したりすることができます。詳細については、[ALTER PIPE](../sql-reference/sql-statements/data-manipulation/ALTER_PIPE.md)、[SUSPEND or RESUME PIPE](../sql-reference/sql-statements/data-manipulation/SUSPEND_or_RESUME_PIPE.md)、[DROP PIPE](../sql-reference/sql-statements/data-manipulation/DROP_PIPE.md)、[SHOW PIPES](../sql-reference/sql-statements/data-manipulation/SHOW_PIPES.md)、および [RETRY FILE](../sql-reference/sql-statements/data-manipulation/RETRY_FILE.md) を参照してください。
